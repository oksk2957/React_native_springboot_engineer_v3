# Plan: DB Refactoring - Problem Table Separation and Statistics Integration

## Status
DRAFT - Awaiting validation

## Overview
Refactor the database schema to separate problem types into dedicated tables while maintaining backward compatibility and integrating statistics management.

## Requirements (confirmed from draft)
- Maintain existing objective random study functionality (problem table)
- Create new `programming_language_problems` table for programming language problems
- Create new `subjective_problems` table for subjective problems
- Migrate all SUBJECTIVE type data from problem to subjective_problems
- Create unified statistics table managing all three problem type lineages
- Support mypage UX: simple stats (attempted, wrong, accuracy) + per-type queries

## Technical Decisions
- Table separation: Dedicated tables (programming_language_problems, subjective_problems)
- Programming language distinction: Single table with `language` field (Java, Python, C, etc.)
- Statistics table: Unified statistics table (extend user_statistics or new)
- Existing data: Full migration (SUBJECTIVE → subjective_problems)
- Objective table: Minor modifications (add problem_type field for unified tracking)

## Current State Analysis

### Schema (from ddl.sql)
**Tables:**
- `subject` (id, name, description, timestamps)
- `users` (id, google_id, email, nickname, role, timestamps)
- `problem` (id, subject_id, question, answer, explanation, type, difficulty, options, is_ai_generated, timestamps)
- `study_session` (id, user_id, subject_id, session_type, status, counts, timestamps)
- `user_answer` (id, user_id, problem_id, session_id, submitted_answer, is_correct, timestamps)
- `user_statistics` (id, user_id, subject_id, counts, last_studied_at, timestamps)
- `wrong_answer_bookmark` (id, user_id, problem_id, bookmarked_at, review_count, last_reviewed_at)

**Problem Types:** SUBJECTIVE, OBJECTIVE (CHECK constraint)

**Key Constraints:**
- problem.type ∈ {SUBJECTIVE, OBJECTIVE}
- problem.difficulty ∈ [1,5]
- Foreign keys with CASCADE delete

### Backend (Spring Boot)
**Current Entities:**
- `Problem` entity with `ProblemType` enum (SUBJECTIVE, FILL_IN_THE_BLANK, OBJECTIVE, MULTIPLE_CHOICE)
- `ProblemRepository`, `SubjectRepository`
- `ProblemQueryMapper.xml` - already using INNER JOIN (no LEFT JOIN found)

**Services:**
- `ProblemService`, `StatisticsService`, `AnswerService`, `UserService`

## Gap Analysis

### Missing from Current Implementation
1. **problem_type field** - Not present in problem entity for unified statistics
2. **language field** - No programming language distinction in problem table
3. **Unified statistics** - Current user_statistics is per-subject, not per-problem-type
4. **Migration scripts** - No scripts to split problem table

## Implementation Steps

### Phase 1: Schema Changes
1. Add `problem_type` column to `problem` table (default: 'OBJECTIVE')
2. Create `subjective_problems` table (inherits from problem structure)
3. Create `programming_language_problems` table (adds `language` field)
4. Update `user_statistics` or create new unified statistics table

### Phase 2: Migration
1. Extract SUBJECTIVE problems to `subjective_problems`
2. Add programming problems to `programming_language_problems`
3. Update `study_session` and `user_answer` with `problem_type` if needed
4. Data validation queries

### Phase 3: Backend Updates
1. Create JPA entities for new tables
2. Update repositories
3. Update services for unified queries
4. Update mappers for per-type statistics

### Phase 4: API Layer
1. Statistics endpoints for per-type queries
2. Problem endpoints updated for new table structure
3. Migration status endpoints
