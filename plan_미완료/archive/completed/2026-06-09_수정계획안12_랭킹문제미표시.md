### Step 2: SQL 쿼리 수정 (#78)
**파일**: `backend/src/main/resources/mapper/MypageStatisticsMapper.xml:155-209`

**수정 전**:
```sql
SUBSTRING(p.question, 1, 60) AS questionText
```

**수정 후**:
```sql
COALESCE(NULLIF(TRIM(SUBSTRING(p.question, 1, 60)), ''), '문제 #' || p.id) AS questionText
```

**理由**:
- `NULLIF(TRIM(...), '')`: 빈 문자열도 NULL로 처리
- `COALESCE(..., '문제 #' || p.id)`: NULL/빈값이면 fallback 표시
- `TRIM()`: 앞뒤 공백 제거

**객관식/주관식/프로그래밍 3곳 모두 적용**

**완료 기준**:
- [ ] 3개 UNION ALL 쿼리 모두 COALESCE 적용
- [ ] 백엔드 컴파일 성공
- [ ] SQL 문법 오류 없음

### Step 3: 프로그래밍 참조 무결성 확인 (#79)
**파일**: `backend/src/main/resources/mapper/MypageStatisticsMapper.xml:192-205`

**현재 쿼리**:
```sql
INNER JOIN programming_language_problems plp ON ua.reference_id = plp.id
WHERE ua.item_type = 'PROGRAMMING_LANGUAGE'
```

**문제**: `user_answer.reference_id`가 `programming_language_problems.id`에 존재하지 않으면 결과 누락

**해결 방안 2가지**:

**방안 A: LEFT JOIN + NULL 필터**
```sql
LEFT JOIN programming_language_problems plp ON ua.reference_id = plp.id
WHERE ua.item_type = 'PROGRAMMING_LANGUAGE'
  AND plp.id IS NOT NULL
```
- 장점: 참조 무결성 깨진 레코드 자동 제외
- 단점: 데이터 손실 (사용자 입장에서 푼 문제가 랭킹에 안나옴)

**방안 B: 참조 테이블 동적 결정 (추천)**
```sql
LEFT JOIN problem p ON ua.item_type = 'OBJECTIVE' AND ua.reference_id = p.id
LEFT JOIN subjective_problems sp ON ua.item_type = 'SUBJECTIVE' AND ua.reference_id = sp.id
LEFT JOIN programming_language_problems plp ON ua.item_type = 'PROGRAMMING_LANGUAGE' AND ua.reference_id = plp.id
```
- 장점: `item_type`과 실제 참조 테이블 불일치해도 처리 가능
- 단점: 쿼리 복잡도 증가

**현재 코드 분석**: 이미 `selectWrongAnswersByDate` (113-153행)에서 LEFT JOIN 패턴 사용 중 → **방안 B 채택**

**수정 작업**:
1. UNION ALL 3개를 단일 쿼리로 재작성
2. LEFT JOIN 3개 + COALESCE로 동적 컬럼 결정
3. `item_type`에 따라 적절한 테이블 참조

**완료 기준**:
- [ ] LEFT JOIN 3개 적용
- [ ] 참조 테이블 불일치해도 결과 표시
- [ ] 백엔드 컴파일 성공

### Step 4: 프론트엔드 fallback 검증 (#78 연관)
**파일**: `InformationExamApp/src/screens/StatisticsScreen.tsx:304`

**현재 코드**:
```tsx
{item.questionText || `문제 #${item.problemId}`}
```

**검증**:
- 백엔드에서 이미 fallback 처리 → 프론트엔드는 추가 수정 불필요
- 다만, `item.questionText`가 빈 문자열(`""`)인 경우 `||` 연산자가 fallback 처리하지 않음

**수정**:
```tsx
{(item.questionText && item.questionText.trim()) || `문제 #${item.problemId}`}
```

**완료 기준**:
- [ ] 빈 문자열 fallback 처리
