## 옵서버(Observer) 패턴 핵심 개념 정리

| 구성 요소 | 설명 |
|-----------|------|
| **주제(Subject)** | 상태를 관리하며, Observer 목록을 유지하고 변경 시 통보 |
| **관찰자(Observer)** | Subject의 상태 변화를 감시하고, 변경 시 업데이트 수행 |
| **구체 주제(ConcreteSubject)** | 실제 상태를 저장하고 Observer 목록 관리 |
| **구체 관찰자(ConcreteObserver)** | Subject의 상태 변화에 반응하여 특정 동작 수행 |

### 주요 특징
- **느슨한 결합(Loose Coupling)**: Subject와 Observer는 서로 독립적으로 변경 가능
- **일대다 관계**: 하나의 Subject에 여러 Observer 등록 가능
- **동적 추가/제거**: 실행 시간에 Observer를 추가하거나 제거 가능
- **MVC 패턴에서 활용**: Model(Subject)과 View(Observer) 분리

## AI 프롬프트 작성 가이드 (랜덤 질문 생성용)

### 프롬프트 작성 핵심 원칙
1. **정답 고정**: 파일명의 마지막 부분(관련정답)이 모든 문제의 고정 정답이 되어야 함
2. **질문 랜덤화**: 같은 정답이라도 질문은 다양한 형태로 랜덤하게 생성
3. **맥락 다양화**: 정의, 특징, 유형, 관계, 예시 등 다양한 맥락에서 질문 생성

### 예시 프롬프트
```
정보처리기사 시험을 위한 주관식 단답형 문제를 생성해주세요.

[핵심 규칙]
- 고정 정답: "옵서버 패턴(Observer Pattern)"
- 모든 문제의 정답은 반드시 "옵서버 패턴(Observer Pattern)"이어야 함
- 질문은 다양한 형태로 랜덤하게 생성

[조건]
- 주제: 디자인 패턴 중 옵서버 패턴(Observer Pattern)
- 문제 유형: 주관식 단답형 (빈칸 채우기)
- 난이도: 중급
- 문제 수: 3개
- 고정 정답: 옵서버 패턴(Observer Pattern) (모든 문제 동일)

[질문 다양화 예시]
- 문제 1: 정의 관점 (일대다 의존 관계를 정의하여 상태 변경 시 자동 통보)
- 문제 2: 구조 관점 (Subject와 Observer의 관계, 발행-구독 메커니즘)
- 문제 3: 활용 관점 (MVC 패턴에서 Model과 View의 분리에 활용)

[출력 형식]
각 문제는 다음 형식으로 작성:
- 질문: 다양한 표현으로 질문
- 정답: 옵서버 패턴(Observer Pattern) (고정)
```

### 프롬프트 작성 시 주의사항
- **정답 고정 확인**: 모든 문제의 정답이 파일명의 관련정답과 일치하는지 검증
- **질문 중복 방지**: 같은 표현의 질문이 반복되지 않도록 다양화
- **맥락 다양성**: 정의, 특징, 유형, 관계, 예시 등 다양한 맥락 활용
- **난이도 조절**: 기출문제 수준의 난이도로 설정
- **오답 유도 방지**: 정답이 명확하고 유일하도록 문제 설계
