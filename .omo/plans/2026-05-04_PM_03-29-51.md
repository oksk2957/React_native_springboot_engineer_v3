## 1. 개요

### 1.1 목적
기존 `problem` 테이블의 구조적 한계를 해결하고, 객관식/주관식/프로그래밍언어 문제를 별도 테이블로 분리하여 데이터 무결성을 확보하며, 통합된 통계 관리 시스템을 구축한다.

### 1.2 범위
- **INCLUDE**: 테이블 분리, 마이그레이션 스크립트, 통계 테이블 설계, 마이페이지 쿼리, 백엔드 엔티티 수정
- **EXCLUDE**: 프론트엔드 UI 구현, 실제 데이터 삽입 (마이그레이션 제외)

## 2. 현재 상태 분석

### 2.1 데이터베이스 현황
- **DB 파일**: `database.db` (SQLite)
- **현재 상태**: 빈 데이터베이스 (테이블 없음)
- **기존 스키마**: PostgreSQL 기반 스키마 (`schema.sql`, `ddl.sql`)
- **이전 시도**: `migration_programming_problems.sql` (problem 테이블에 프로그래밍 문제 통합 시도 - 구조적 한계 발견)

### 2.2 기존 테이블 구조 (schema.sql 기준)
```sql
-- problem 테이블 (객관식 + 주관식 혼재)
CREATE TABLE problem (
    id bigserial NOT NULL,
    subject_id int8 NOT NULL,
    question text NOT NULL,
    answer text NOT NULL,
    explanation text NULL,
    type text NOT NULL,  -- 'SUBJECTIVE' | 'OBJECTIVE'
    difficulty int4 NULL,
    option1~5 text NULL,  -- 객관식 전용 필드
    is_ai_generated bool DEFAULT false NULL,
    created_at timestamptz DEFAULT now() NULL,
    updated_at timestamptz DEFAULT now() NULL
);
```

### 2.3 문제점
1. **데이터 구조 혼재**: 객관식 전용 필드(option1~5)와 주관식 데이터가 동일 테이블에 존재
2. **프로그래밍언어 문제**: 별도 필드(language 등)가 없어 단일 테이블에 통합 불가
3. **통계 관리**: 3개 줄기(객관식/주관식/프로그래밍) 통합 통계 관리 어려움
4. **확장성**: 새로운 문제 유형 추가 시 테이블 구조 변경 필요

## 3. 신규 테이블 설계

### 3.1 테이블 분리 전략
- **객관식**: 기존 `problem` 테이블 유지 (OBJECTIVE 타입만)
- **주관식**: 신규 `subjective_problems` 테이블
- **프로그래밍언어**: 신규 `programming_language_problems` 테이블 (language 필드로 구분)

### 3.2 신규 스키마 정의

#### 3.2.1 객관식 문제 테이블 (기존 problem 테이블 수정)
```sql
CREATE TABLE problem (
    id bigserial NOT NULL,
    subject_id int8 NOT NULL,
    question text NOT NULL,
    answer text NOT NULL,
    explanation text NULL,
    difficulty int4 NULL,
    option1 text NOT NULL,
    option2 text NOT NULL,
    option3 text NULL,
    option4 text NULL,
    option5 text NULL,
    is_ai_generated bool DEFAULT false NULL,
    created_at timestamptz DEFAULT now() NULL,
    updated_at timestamptz DEFAULT now() NULL,
    CONSTRAINT problem_difficulty_check CHECK (((difficulty >= 1) AND (difficulty <= 5))),
    CONSTRAINT problem_pkey PRIMARY KEY (id),
    CONSTRAINT problem_subject_id_fkey FOREIGN KEY (subject_id) REFERENCES subject(id) ON DELETE CASCADE
);
-- type 필드 제거 (객관식 전용이므로)
```

#### 3.2.2 주관식 문제 테이블 (신규)
```sql
CREATE TABLE subjective_problems (
    id bigserial NOT NULL,
    subject_id int8 NOT NULL,
    question text NOT NULL,
    answer text NOT NULL,
    explanation text NULL,
    difficulty int4 NULL,
    keywords text NULL,
    is_ai_generated bool DEFAULT false NULL,
    created_at timestamptz DEFAULT now() NULL,
    updated_at timestamptz DEFAULT now() NULL,
    CONSTRAINT subjective_problems_difficulty_check CHECK (((difficulty >= 1) AND (difficulty <= 5))),
    CONSTRAINT subjective_problems_pkey PRIMARY KEY (id),
    CONSTRAINT subjective_problems_subject_id_fkey FOREIGN KEY (subject_id) REFERENCES subject(id) ON DELETE CASCADE
);
```

#### 3.2.3 프로그래밍언어 문제 테이블 (신규)
```sql
CREATE TABLE programming_language_problems (
    id bigserial NOT NULL,
    subject_id int8 NOT NULL,
    language text NOT NULL,
    question text NOT NULL,
    answer text NOT NULL,
    explanation text NULL,
    difficulty int4 NULL,
    code_snippet text NULL,
    expected_output text NULL,
    keywords text NULL,
    is_ai_generated bool DEFAULT false NULL,
    created_at timestamptz DEFAULT now() NULL,
    updated_at timestamptz DEFAULT now() NULL,
    CONSTRAINT programming_language_problems_difficulty_check CHECK (((difficulty >= 1) AND (difficulty <= 5))),
    CONSTRAINT programming_language_problems_pkey PRIMARY KEY (id),
    CONSTRAINT programming_language_problems_subject_id_fkey FOREIGN KEY (subject_id) REFERENCES subject(id) ON DELETE CASCADE
);
```

#### 3.2.4 통합 통계 테이블 (신규)
```sql
CREATE TABLE integrated_statistics (
    id bigserial NOT NULL,
    user_id int8 NOT NULL,
    problem_type text NOT NULL,
    reference_id int8 NOT NULL,
    subject_id int8 NOT NULL,
    is_correct bool NOT NULL,
    attempted_at timestamptz DEFAULT now() NULL,
    time_spent_seconds int4 NULL,
    created_at timestamptz DEFAULT now() NULL,
    CONSTRAINT integrated_statistics_pkey PRIMARY KEY (id),
    CONSTRAINT integrated_statistics_user_id_fkey FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT integrated_statistics_subject_id_fkey FOREIGN KEY (subject_id) REFERENCES subject(id) ON DELETE CASCADE,
    CONSTRAINT integrated_statistics_problem_type_check CHECK ((problem_type = ANY (ARRAY['OBJECTIVE'::text, 'SUBJECTIVE'::text, 'PROGRAMMING'::text])))
);

CREATE INDEX idx_integrated_statistics_user ON integrated_statistics(user_id);
CREATE INDEX idx_integrated_statistics_type_ref ON integrated_statistics(problem_type, reference_id);
```

#### 3.2.5 기존 테이블 수정 (study_session, user_answer, user_statistics, wrong_answer_bookmark)

## 4. 마이그레이션 전략

### 4.1 데이터 마이그레이션 계획

#### 4.1.1 기존 SUBJECTIVE 데이터 마이그레이션
```sql
INSERT INTO subjective_problems (subject_id, question, answer, explanation, difficulty, is_ai_generated, created_at, updated_at)
SELECT subject_id, question, answer, explanation, difficulty, is_ai_generated, created_at, updated_at
FROM problem
WHERE type = 'SUBJECTIVE';
```

#### 4.1.2 기존 OBJECTIVE 데이터 정리
```sql
DELETE FROM problem WHERE type = 'SUBJECTIVE';
ALTER TABLE problem DROP COLUMN type;
```

#### 4.1.3 프로그래밍언어 데이터 초기화
```sql
INSERT INTO subject (name, description) 
VALUES ('프로그래밍언어', 'C, Java, Python 등 언어 특성')
ON CONFLICT (name) DO NOTHING;
```

### 4.2 마이그레이션 순서
1. 신규 테이블 생성 (`subjective_problems`, `programming_language_problems`, `integrated_statistics`)
2. 기존 테이블에 필드 추가 (problem_type, reference_id)
3. SUBJECTIVE 데이터 이관
4. 제약조건 수정 (외래키 제거, problem_type 체크 추가)
5. problem 테이블 정리 (SUBJECTIVE 데이터 삭제, type 필드 제거)
6. 인덱스 생성

## 6. 마이페이지 통계 조회 설계

### 6.1 간단 통계 (메인 대시보드)
```json
{
  "totalProblemsSolved": 150,
  "totalCorrect": 120,
  "totalIncorrect": 30,
  "correctRate": 80.0,
  "byType": {
    "OBJECTIVE": {"solved": 80, "correct": 70, "rate": 87.5},
    "SUBJECTIVE": {"solved": 40, "correct": 30, "rate": 75.0},
    "PROGRAMMING": {"solved": 30, "correct": 20, "rate": 66.7}
  }
}
```

### 6.2 통계 조회 쿼리
```sql
SELECT 
    problem_type,
    COUNT(*) as total_attempted,
    SUM(CASE WHEN is_correct THEN 1 ELSE 0 END) as correct_count,
    ROUND(SUM(CASE WHEN is_correct THEN 1 ELSE 0 END) * 100.0 / COUNT(*), 2) as correct_rate
FROM integrated_statistics
WHERE user_id = ?
GROUP BY problem_type;
```
