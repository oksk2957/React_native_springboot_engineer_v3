## 📋 개요

- **영향 범위**: 모든 인증 의존 기능 (통계, 랭킹, 오답 노트)

## 🔍 문제 분석

### 1. 로그인 400 에러
**현상**: 서버 로그에서 동일한 시간에 2개의 요청 발생
- `e14f3d32`: 200 OK (정상)
- `f4d20c6a`: 400 Bad Request (실패)

**원인**: `onAuthStateChange`가 두 이벤트에서 발동
- `INITIAL_SESSION`: 앱 시작 시 (JWT 없으면 400)
- `SIGNED_IN`: OAuth 완료 시 (정상)

### 2. 통계/랭킹/오답이 안 나옴
**원인**: 400 에러로 JWT 발급 실패 → `Authorization` 헤더 없음 → `userId == null` → 빈 배열 반환

## 🎯 해결 방안

### 수정 1: `onAuthStateChange` 이벤트 필터링
**파일**: `InformationExamApp/App.tsx` (220-319번 줄)

**변경 전**:
```typescript
const { data: authListener } = supabase.auth.onAuthStateChange(async (event, session) => {
  console.log('[App] Supabase auth 상태 변경:', event, session?.user?.email ?? 'no user');

  if (event === 'SIGNED_IN' && session?.access_token) {
    console.log('[App] SIGNED_IN 이벤트 감지 - 백엔드 JWT 교환 시작');
    // ... 백엔드 JWT 교환 로직
  }
});
```

**변경 후**:
```typescript
const { data: authListener } = supabase.auth.onAuthStateChange(async (event, session) => {
  console.log('[App] Supabase auth 상태 변경:', event, session?.user?.email ?? 'no user');

  // DEBUG: [Auth-Fix-2026-06-07] INITIAL_SESSION 이벤트 무시
  // 원인: INITIAL_SESSION에서 loginWithGoogle 호출 → 400 에러
  // 해결: INITIAL_SESSION은 restoreSessionFromJWT가 처리하므로 무시
  if (event === 'INITIAL_SESSION') {
    console.log('[App] INITIAL_SESSION 무시 - restoreSessionFromJWT가 처리');
    return;
  }

  if (event === 'SIGNED_IN' && session?.access_token) {
    console.log('[App] SIGNED_IN 이벤트 감지 - 백엔드 JWT 교환 시작');
    // ... 기존 로직 유지
  }
});
```

**효과**: 
- 경로 A (`restoreSessionFromJWT`): 앱 재시작 시 자동 로그인
- 경로 B (`SIGNED_IN`): 새 OAuth 로그인만 처리
- 두 경로가 명확히 분리되어 충돌 제거

### 수정 2: Supabase JWT Secret 환경변수 설정
**파일**: `backend/.env` (또는 `backend/src/main/resources/application.properties`)

**변경 전**:
```properties
supabase.jwt-secret=${SUPABASE_JWT_SECRET:eyJhbGciOiJIUzI1NiIs...}  ← anon key 기본값
```

**변경 후**:
```properties
supabase.jwt-secret=${SUPABASE_JWT_SECRET}  ← 기본값 제거, 환경변수 필수
```

**작업**:
1. Supabase 대시보드에서 JWT Secret 확인
   - Settings → API → JWT Settings → JWT Secret
2. `.env` 파일에 추가:
   ```
   SUPABASE_JWT_SECRET=<실제 JWT Secret 값>
   ```

**효과**: JWT 서명 검증이 정상적으로 동작

## 🚨 리스크 및 대응

### 리스크 1: JWT Secret 확인 불가
**대응**: Supabase 대시보드 접근 불가 시
- 임시로 `supabase.jwt-secret` 검증 로직 비활성화

**대응**: `restoreSessionFromJWT`가 정상 동작하는지 확인
- AsyncStorage에서 JWT 읽기 성공하는지 로그 확인
- JWT 만료 시간 확인 (12시간)

- Settings → API → Redirect URLs
- `http://localhost:9000/auth-callback` 또는 `exp://...` 형식
